# REVIEW-2026-07-16 — Playtest feedback triage (v1.6 scoping)

Source: user playtest notes (2026-07-16), a full-run balance/feel review of shipped **v1.5.0**
(save version 10). Method: four parallel `Explore` subagents (skills/TP, stats/combat, economy/
items, classes/XP/locations) + lead core reads (`character.js`, `battle.js`, `balance.js`) +
`reference/` mining. Every finding below is grounded in a code citation; every proposed fix
carries a `[archived]/[invented]/[revised]` tag per CLAUDE.md cardinal rule 1.

> **Ratchet-principle note (LEAD-PLAYBOOK §0.3).** The playbook freezes *shipped* balance and
> says to fit new mechanics to it. This release is the deliberate exception: the user is
> directing a re-tune of shipped constants *because the shipped feel is wrong*. User-directed
> overrides of shipped values are tagged `[revised]`. Every combat-math change is still
> sim-gated against the difficulty contract before it locks (P0).

Severity: **H** breaks a core loop or the difficulty contract · **M** notable friction ·
**L** polish. "Sim" = must pass a `/balance-sim` gate before locking.

## A. Progression — XP / skills / classes / Training Points

| ID | Feedback | Root cause (cite) | Sev | Proposed fix (tag) | Sim |
|----|----------|-------------------|-----|--------------------|-----|
| PG‑1 | "XP gain is very high" / leveling too fast | Flat **~3.5–5 kills/level across all of 1–100**, ~2 kills at the very start; `XP_TO_LEVEL=50·(n−1)^1.8` vs `MONSTER_XP=15+10·lvl` — linear reward outpaces the ~n^0.8 marginal curve ([balance.js:20](../js/balance.js#L20),[:99](../js/balance.js#L99)); **Fury XP bonus is uncapped** (`+1%/tick`, no ceiling — [battle.js:1318](../js/core/battle.js#L1318)); quest XP dumps up to 5000 ([quests.js](../js/data/quests.js)) | H | Steepen the curve (raise exponent / add a linear term) toward the accepted "L30 ≈ 3–4h" anchor; **cap the Fury XP bonus** (ratchet precedent: Frenzied capped +40%). `[revised]` | ✔ |
| PG‑2 | "Class leveling super fast — all abilities in a few combats" | Class XP = full battle XP (primary) / half (secondary) but the class curve `30·(n−1)^1.6` caps at ~lvl 8–13; a tier‑3 class fully unlocks in **~2–3 kills**, tier‑1 in ~13 ([classes.js:220](../js/core/classes.js#L220),[:229](../js/core/classes.js#L229)) | H | Steepen class‑XP curve and/or scale ability point‑costs so a class spans a meaningful slice of its tier band; higher tiers should not unlock *faster* in absolute kills. `[revised]` | ✔ |
| PG‑3 | "Skills level far too slow, XP doesn't scale, L100 plays with tier‑2 abilities" | Skill XP is **flat 8/win** per qualifying skill, never scales up, decays to 1 when over‑levelled ([character.js:401](../js/core/character.js#L401),[battle.js:1358](../js/core/battle.js#L1358)); ~3,200 wins to reach skill 50. Worse: weapon/armor skill scaling **caps at skill ~7–8** (`WEAPON_SKILL_DAMAGE_CAP 0.10`, `ARMOR_SKILL_ARMOR_CAP 0.15`) so 90% of the `2·lvl+1` skill range buys **zero** benefit | H | Scale `SKILL_XP_PER_USE` with monster level (base is invented; only the *decline* is archived); **raise/relax the weapon+armor scaling caps** so high skill keeps paying; consider per‑use (not per‑win) award for weapon/armor feel. `[revised]` | ✔ |
| PG‑4 | "Tons of TP, nothing to spend it on" | Only sink is Academy techs; ~258 TP earned by L100 vs a focused build spending ~160 → ~100 idle; and high‑rank `skillReq` (up to 70) means you often *can't* spend it while skills lag ([world.js:572](../js/core/world.js#L572), agent report) | M | Couples with PG‑3 (unblock spending). Add TP sinks (respec, more ranks) or trim TP/level; primarily solved by fixing skill pacing. `[invented]` | – |

## B. Combat & Stats

| ID | Feedback | Root cause (cite) | Sev | Proposed fix (tag) | Sim |
|----|----------|-------------------|-----|--------------------|-----|
| CB‑1 | "Defense too strong — takes almost no damage with light armor; endurance makes non‑bosses trivial" | Mitigation is **flat subtraction with a floor of 1**: `dmg = max(1, raw − armor·fear)` ([battle.js:549](../js/core/battle.js#L549)); Endurance→armor is **1:1** ([character.js:313](../js/core/character.js#L313)) and equipped armor stacks additively on top, so a mid‑level char's armor exceeds a same‑level monster's `3+2·lvl` hit → most hits floor to **1** | H | **Penetration floor**: a hit always lands ≥ *k*% of its rolled raw damage regardless of armor (structural), and/or drop Endurance→armor below 1:1. **Contract‑touching — user decision (Q2).** `[revised]/[invented]` | ✔ |
| CB‑2 | "Magic skills don't scale" | Spell damage = `tech.power·(1+INT·0.02)` — the governing school level is **never read** ([battle.js:792](../js/core/battle.js#L792),[:967](../js/core/battle.js#L967)); school level only gates learning ranks + banks XP | H | (a) School level → **+% spell power** (parallel to weapon skills); (b) **INT boosts magic‑skill XP gain** — `[archived]` [Intelligence.md](../reference/manual/Intelligence.md) ("Increases the Experience gained in … Rods, Evocation, Conjuration, Alteration, Absorption, Abjuration"), currently unimplemented. `[archived]+[invented]` | ✔ |
| CB‑3 | "Attack out‑scales magic abilities" | Basic attack = `STR/2.5 + weaponBonus` where the **bought‑weapon term dominates** and scales with gold (13→23→53→73→92); spells cost 7× energy, can miss (INT hit‑roll), and are mitigated by magic‑armor ([character.js:281](../js/core/character.js#L281) vs [battle.js:936](../js/core/battle.js#L936)). Cause is exactly the user's: too much gold + inert magic skills + no magic scaling | H | Resolved by CB‑2 + EI‑1 (curb gold/weapon dominance) + PG‑3 (skills matter). Give class/skill abilities a real scaling term. `[revised]` | ✔ |
| CB‑4 | "Rods are just a melee weapon without skills" | A Rod is a plain melee weapon whose only quirk is INT‑scaled damage ([character.js:269](../js/core/character.js#L269)); **zero** rod handling in battle.js; no spell‑power boost, no energy discount, no Rod tech chain | M | Give Rods a caster identity: equipping a Rod boosts spell power and/or cuts tech energy cost, and/or add a Rod weaponTech chain. `[invented]` (Rods←Int is archived; the boost is invented) | ✔ |
| CB‑5 | "INT does very much (magic def + magic dmg + energy + rod dmg + accuracy)" | INT feeds **4** roles: magic‑armor 1:1 ([character.js:323](../js/core/character.js#L323)), spell damage, Rod melee, spell hit‑chance. **Energy is level‑based, not INT** ([character.js:249](../js/core/character.js#L249)) — that item in the notes is inaccurate | M | Design call: keep INT as the caster mono‑stat but ensure parity, or move one role (e.g. accuracy) off it. Lower priority; revisit after CB‑1/CB‑2. `[invented]` | – |
| CB‑6 | "Weight scaling from STR goes very hard" | Carry capacity = `strength·10` with **no base term** ([inventory.js:24](../js/core/inventory.js#L24)); STR 5 = 50 cap, punishing any non‑STR build | M | Add a base capacity term + gentler slope so casters/dex builds aren't encumbered. `[invented]` | – |

## Proposed v1.6 shape (pending the Q1–Q3 decisions below)

Phased, each phase green on all ten suites before commit; P0 is the mandatory sim gate.

- **P0 — sim gate.** Lock the combat‑math changes (CB‑1 penetration floor, CB‑2 magic scaling,
  PG‑3 skill scaling, EI‑6 camp sustain) against the difficulty contract before any game code
  is written — the F1/E0/v1.4‑P0 precedent.
- **P1 — Combat & Stats** (CB‑1..CB‑6).
- **P2 — Progression** (PG‑1..PG‑4).
- **P3 — Economy & Items** (EI‑1..EI‑7).
- **P4 — Content & flow** (CF‑1..CF‑3).
- **P5 — docs + release** (changelog/README/save‑version/suites/artifact; deploy user‑gated).

Save version: most of this is constants + data + formula changes → **likely stays 10**; only a
new persisted character field (e.g. a respec ledger) would force a bump. TBD at spec time.
